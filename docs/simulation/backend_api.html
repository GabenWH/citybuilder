Simulation Backend API

Binary buffer interface for Unity rendering. JSON is load-time only.

Overview | Backend API | Modes

Runtime Data Flow

Startup (one-time)
1) Unity sends network snapshot + seed data
2) Backend builds internal structures

Simulation loop (backend)
1) Step(dt)
2) Update lifecycle and demand
3) Update per-mode movement
4) Produce render buffers

Unity loop
1) Pull render buffers
2) Update instances + UI

Core Messages

Inputs to Backend

  • NetworkSnapshot (roads, rails, stops, transfers)
  • AgentSeedBatch (initial population)
  • Commands (spawn, despawn, reroute, pause)
  • SimParams (timestep, mode weights)

Outputs to Unity

  • RenderBuffer
  • Stats (throughput, congestion, averages)
  • Debug buffers (optional)

Binary Buffer Layout (v1)

These are GPU-friendly structs. Unity reads them as raw bytes.

struct RenderInstance
{
    float px, py, pz;   // position
    float fx, fy, fz;   // forward
    uint  id;           // stable id for selection/debug
    uint  mode;         // 0 car, 1 bus, 2 train, 3 metro, 4 plane, 5 walk
};

struct SimStats
{
    uint active_agents;
    uint traveling_agents;
    float avg_travel_time;
    float avg_speed;
};

Render buffers are per mode, fixed capacity, and updated every frame.

Network Snapshot (load-time JSON)

JSON is used only at startup or for offline authoring.

{
  "nodes": [...],
  "segments": [...],
  "lanes": [...],
  "stops": [...],
  "transfers": [...]
}

Runtime uses binary buffers; no JSON parsing per frame.

Lifecycle API (simple v1)

enum AgentState
{
    AtHome,
    TravelingToDest,
    AtDest,
    TravelingHome
};

struct Agent
{
    uint id;
    uint home_id;
    uint dest_id;
    uint state;
    uint mode;
    float eta;
};

Lifecycle is backend-only, extendable later with schedules and needs.

Update Order (per tick)

  1. Advance time, process timers.
  2. Transition lifecycle states (depart/arrive).
  3. Route or re-route (if needed).
  4. Mode simulation (roads, rails, walk, air).
  5. Emit render buffers and stats.

Notes on Performance

  • Unity does zero per-agent logic.
  • Agents can be sampled for rendering (visuals != full sim).
  • All buffers are fixed-size to avoid allocations.
  • Backend can run faster than Unity and interpolate results.
