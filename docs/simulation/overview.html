City Builder Simulation Overview

Backend-first architecture with Unity as renderer. Simple lifecycle now, extensible later.

Overview | Backend API | Modes

Goals

  • Support huge populations (500k to 5M) with realistic-enough movement.
  • Keep lifecycle logic intact but decouple it from movement fidelity.
  • Allow multiple transit modes: car, bus, train, metro, plane, walking.
  • Use Unity for rendering and tools, not for core simulation.

Core Architecture

Backend (authoritative)

  • Lifecycle and demand generation.
  • Routing and traffic simulation.
  • Multimodal networks + transfers.
  • Deterministic time-stepping.

Unity (renderer/tools)

  • Send inputs (network snapshot, seeds, actions).
  • Receive render buffers (positions, headings, mode ids).
  • Visualize and debug.

Unity can still provide editor tooling, but core sim data lives in the backend.

Lifecycle (v1, simple)

Lifecycle runs in the backend and is intentionally minimal for now. It is easy to extend later.

State machine:
AtHome -> TravelingToDest -> AtDest -> TravelingHome -> AtHome

Each agent has:
- home_id
- primary_dest_id
- state
- next_state_time
- selected_mode
- eta

Future: schedules, needs, multi-stop chains, stochastic deviations.

Movement Model

Flow-based traffic for scale. We simulate lane cells, not per-agent collision.

  • Road lanes are 1D grids of fixed-size cells.
  • Cells store occupancy, speed, and next-step intent.
  • Intersections use simple right-of-way or phase rules.
  • Visual agents are sampled from occupancy.

This preserves "flowing cars" visuals without per-agent micro-sim costs.

Modes and Transfers

  • Car/Bus: lane-cell flow.
  • Train/Metro: schedule-based segments.
  • Plane: coarse OD + airport nodes.
  • Walking: coarse grid or sidewalk graph.
  • Transfers: walk links between modes (stops, stations, airports).

Unity/Backend Interface

Unity does not read a JSON 30 times a second.

  • JSON is only for initial load or offline data.
  • At runtime, we use binary buffers or shared memory.
  • Unity reads a compact buffer each frame (positions, headings, ids).
Typical runtime flow:
1) Unity sends initial network + seed data once
2) Backend runs Step(dt) on its own loop
3) Unity pulls render buffers each frame

Extensibility Notes

  • Add needs/schedules without changing movement model.
  • Add new transit modes by adding a network + transfer links.
  • Swap out routing algorithm without touching Unity.
  • Scale population by adjusting flow density and visual sampling.
